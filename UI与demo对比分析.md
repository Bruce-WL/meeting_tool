# 会议纪要 UI 与 demo PDF 对比分析

## 说明

- **需求描述 prd.md** 当前为空，无法从 PRD 核对需求。
- 对比依据：**会议纪要_demo.pdf** 的版面与内容结构。
- 结论：当前 UI **未**与 demo PDF 对应，存在多处结构和展示差异。

---

## 一、Demo PDF 中的版面与结构（目标效果）

### 1. 顶部区域（Header）

| 元素 | Demo 中的表现 |
|------|----------------|
| 主标题 | **智能纪要：**2026年01月27日 19点25分 2026年1月27日 |
| 会议主题 | 2026年01月27日 19点25分 |
| 上传时间 | 1月27号（周二）20:28（GMT+08） |
| 免责声明 | 智能纪要由 **AI** 生成，可能不准确，请谨慎甄别后使用 |

**特点**：强调「智能纪要 + 日期时间」、会议主题、上传时间、AI 免责声明；**没有**「参会人」「会议时长」等字段。

### 2. 总结（Summary）

- **形式**：一整块**连贯的总结正文**，不是多张卡片。
- **内容**：首段总述 + **多级列表**：
  - 一级：`•`（如「2025年工作回顾与感受」）
  - 二级：`◦`（如「整体感受」「AI行业发展主线」）
  - 三级：`▪`（如「焦虑与变化并存：…」）
- **排版**：长文流式阅读，可含小标题（如「2025年AI行业发展主线分析」「2026年核心发展方向规划」）。

### 3. 待办（Todo）

- **形式**：简单列表，每条一行文字。
- **示例**：`组织手机部讨论 OS 如何为 AI 做基建`
- **特点**：无勾选状态、无负责人（assignee）、无「卡片化」装饰。

### 4. 智能章节（Smart Chapters）

- **形式**：每条 = **时间戳** + **章节标题** + **章节描述段落**。
- **示例**：`00:01` → 「回顾2025年感受与变化，展望2026年更开心」+ 一段说明文字。
- **特点**：时间在左，标题与正文在下/右侧，偏文档式列表，不是「时间轴圆点 + 播放按钮」的强交互样式。

### 5. 关键决策（Key Decisions）

- **结构**：
  - **关键决策** 主块：含「**问题**」「**讨论方案**」「**决策依据**」等结构化字段。
  - **其他决策**：a. b. c. d. e. 的列表。
- **当前实现**：仅为 `string[]`，没有「问题 / 讨论方案 / 决策依据」的层级，也没有「其他决策」的 a/b/c 列表形式。

### 6. 金句时刻（Golden Moments）

- **单条格式**：`「金句内容」 — 说话人X，说明文字。`
- **示例**：`「2025年是'乱花渐欲迷人眼'…」 — 说话人1，形象地总结了…`
- **当前实现**：`string[]`，未拆分为「引用」「说话人」「说明」三部分，展示形式也与 demo 不一致。

### 7. 相关链接（Related Links）

- **Demo 中有**：
  - 妙记：2026年01月27日 19点25分（链接）
  - 文字记录：2026年01月27日 19点25分（链接）
- **当前项目**：**完全缺少**「相关链接」区块。

## 二、当前项目实现情况

### 已实现

| 模块 | 实现方式 | 与 Demo 的差异 |
|------|----------|----------------|
| Header | 标题 + 日期 + 时长 + 参会人（图标） | 无「智能纪要」前缀、无会议主题/上传时间/免责声明；多了时长、参会人 |
| Summary | **多张卡片**（如产品目标/技术架构/UI规范），每卡多个 points | Demo 是**一段长总结 + 多级列表**，不是多卡片 |
| Todo | 带勾选、assignee 的待办列表 | Demo 是简单文字列表，无勾选、无负责人 |
| Smart Chapters | 时间轴 + 圆点 + 播放按钮 + 时间戳 + 标题 + 摘要 | 信息有，但视觉和交互偏「时间轴/播放」，Demo 偏文档列表 |
| 关键决策 | InfoBlock，展示 `string[]` | 无「问题/讨论方案/决策依据」结构，无「其他决策」a/b/c |
| 金句时刻 | InfoBlock，展示 `string[]` | 无「引用 — 说话人，说明」的拆分与版式 |

### 未实现

- **相关链接**：整块缺失。
- **总结**：按 Demo 应为「单块长文 + 多级列表」，而不是多张 summary 卡片。

### 数据与 Demo 内容不一致

- `meetingData.ts` 为自编的「产品需求评审会议」示例（Alex/Bob/Charlie、React/Tailwind 等）。
- Demo PDF 为真实的「智能纪要」示例（2026年1月27日、小爱团队、人车家战略等）。
- 若要以 Demo 为基准做 UI 对齐，需要至少有一套与 PDF 一致的**示例数据**（或从 PDF 抽取）用于页面展示。

## 三、差异汇总（按优先级）

### 高优先级（结构/版式与 Demo 明显不符）

1. **Header**：改为「智能纪要：日期时间」+ 会议主题 + 上传时间 + AI 免责声明；去掉或弱化「时长」「参会人」若 Demo 中未强调。
2. **总结**：从「多张卡片」改为「**单块总结正文 + 多级列表**」（• ◦ ▪），并支持长段落与子标题。
3. **关键决策**：支持结构化（问题、讨论方案、决策依据）+「其他决策」的 a/b/c/d/e 列表展示。
4. **金句时刻**：支持「引用」「说话人」「说明」三段式展示，版式与 Demo 一致。
5. **相关链接**：新增「相关链接」区块，至少包含「妙记」「文字记录」等链接。

### 中优先级（展示形式可更贴近 Demo）

6. **待办**：若严格对齐 Demo，可改为简单列表（无勾选、无 assignee）；或保留现有能力但提供「简洁模式」。
7. **智能章节**：可调整为更接近「文档列表」（时间戳 + 标题 + 段落），弱化时间轴/播放按钮风格。

### 低优先级（数据与视觉）

8. **示例数据**：用与 Demo PDF 一致或高度一致的内容做 `meetingData`（或单独 demo 数据文件），便于对照验收。
9. **字体/间距/色块**：若需像素级还原，再对照 PDF 调字体、行距、分区背景等。
